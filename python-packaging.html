From Script to Project part 2. - Packaging Your Code in Python

Posted by Bob on Sat 01 July 2017 in Learning • 4 min read

This week's article is about packaging your Python code. It can be daunting at first but it is actually pretty simple.

Last week we introduced Karma Bot in part 1 of this series. I will use it to show you how I ended up organizing the code. Then I will show a simpler script which I refactored yesterday into a package.

Packaging your code makes it easier for others to use. It also adds more structure to your code which makes it more maintainable. Finally it mamespaces your code, "one honking great idea" as the Zen puts it.

At the very basic level you create a package by putting one or more modules (.py files) inside a folder together with a __init__.py file. The __init__.py file turns the folder into a package. Your code should ideally not go in that file. It is used for imports and setup.

Example 1 - Karma bot

At this moment we have two packages in karmabot:

$ ls bot
__init__.py    karma.py    slack.py

$ ls utils
__init__.py    get_botid.py

And main.py in the top level directory is the driving script:

$ ls
...
main.py
...

There is not one best way to structure you code. Looking at above I might actually refactor this to add slack.py and get_botid.py into a slack package. This is work in progress.

I tried to limit init to imports and setup. I might move things around as we go, but this is far better than the first version I had. This had all code in one big file. Unfortunately this was before my first commit so cannot show it.

Example 2 - Twitter Archive Stats

That's why I have another example lined up. It's a smaller script so better to demo.

As you see all the code is lumped together in one file. There is also way too much going on under if __name__ == '__main__':. This is not code we can re-use. Of course most scripts start like this. But if you don't step back every now and then, it becomes a mess really quickly.

A great book on refactoring I read this year is Martin Fowler's Refactoring. The code samples are Java, but easy to follow. More importantly the concepts are generic you will fully benefit as a Python programmer.

Packaging to the rescue! Here are the steps:

  • First. Don't write any code yet. Think about the various things this script tries to accomplish. What are the main responsibilities? In this case we:

    1. Grab tweets.csv we obtained from Twitter and parse the data out
    2. We Parse the tweets counting certain objects
    3. We print the results to stdout

  • So we have 3 things it does. As it is a small script I will just make one package. I created a folder called "archive" with in it:

    • A module = Python script file for each functionality
    • An __init__.py file that makes it into a package

  • Then I started moving code around. The nice thing I noted that this led to more refactoring! For example the for row in data: loop was 25 lines, now it's only 15 lines, 4 of which are blank. I used the extract method refactoring technique for this. Having additional helper methods makes it also more readable.

Refactoring your code is a really positive side effect of packaging!

  • One note on imports. Starting off with:

    $ ls
archive    main.py

    Adding the init file to the archive package, in main.py I could import like this:

        from archive.report import print_header, print_results
    ...

    It's common to make this shorter by adding imports to the init file, like so:

    $ more archive/__init__.py
from .tweets import parse_csv
from .stats import calc_stats
from .report import print_header, print_results

    Now I can just import from the archive package, adding them to one import statement:

    from archive import parse_csv, calc_stats, print_header, print_results

See also Be Pythonic: __init__.py
